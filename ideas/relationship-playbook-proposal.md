# Relationship Playbook AI — 项目 Proposal

**日期**：2026-02-24
**作者**：Indie 独立开发者
**项目状态**：需求分析完成，进入 MVP 开发阶段

---

## 1. 项目起源

这个项目起源于一个真实的痛点：

> 和一个重要的人谈完一次，心里知道"没谈好"，或者"下次还要再谈"——但脑子里只有一堆乱的印象，不知道哪里出了问题，下次见面还是会重蹈覆辙。

现有的解法要么是找朋友倾诉（主观、偏情绪），要么是自己写日记（没有结构、写完不知道怎么用），要么是什么都不做。

没有一个工具能回答这个问题：**下次见面，我该怎么说？**

---

## 2. 项目目的

### 2.1 产品目的

用AI把非结构化的人际记录（复盘笔记、聊天记录）转化成可执行的沟通规则（Playbook）。

**一句话愿景：**

> AI 不只是帮你记住发生了什么，而是告诉你下次该怎么做。

### 2.2 关键产品理念：输出是 Playbook，不是 Summary

这是与市场上所有现有产品最核心的差异：

| 现有产品 | 本产品 |
|---|---|
| "这个人是 INTJ 型人格" | "下次先给同行案例，再提方案" |
| "你们关系处于磨合期" | "不要问他顾虑是什么，他不会直接说" |
| "他是数据驱动型决策者" | "这两个方向你倾向哪个？我按你的意思调整。" |

输出不是"解释他是谁"，而是"你该怎么做"。

### 2.3 开发者目的

作为 indie 开发者，这个项目的目标是：

- **建立技术社区的 visibility**（GitHub、Hacker News、X/Twitter）
- 探索一个有价值的 AI 产品方向：结构化规则提取 + 可执行行动建议
- 为之后的迭代（更大众化的方向）打下基础

**不以盈利为第一目标**，但需要做到有趣、有技术讨论价值、可传播。

## 3. 市场与竞品分析

### 3.1 三个相邻赛道

**Personal CRM**（Notion、Clay 等）
- 主要功能：记录联系人、提醒跟进
- AI 的角色：摘要、提醒
- 缺失：没有"下次怎么说"

**B2B 销售人格洞察**（CrystalKnows、Humantic AI）
- 主要功能：基于人格框架（DiSC）输出沟通建议
- 优点：有具体的行动建议
- 缺失：依赖公开信息（LinkedIn），无法处理私人复盘；闭源 SaaS

**聊天记录分析**（Mosaic Chats、Relationship AI: Chat Analyzer）
- 主要功能：上传聊天记录，输出洞察与可分享卡片
- 优点：易传播（viral）
- 缺失：输出是"关系洞察"，不是"可执行规则"

### 3.2 本产品的差异化定位

> 竞品输出"你们关系怎样"，本产品输出"你下次该怎么说"。

另一个关键差异：**开源 + 本地部署**。竞品全部是闭源 SaaS，数据上传到第三方服务器。本产品开源、数据不离开用户浏览器，这对有隐私顾虑的职场用户是真实的差异化。

## 4. 目标用户

### 4.1 三类潜在用户

**第一类：职场关键关系维护者**
- 典型场景：搞定强势老板、推动跨部门项目、维护大客户关系
- 痛点特征：这段关系的质量直接影响工作结果，失误成本高，愿意花时间复盘
- 付费意愿：高

**第二类：独立顾问 / 创业者**
- 典型场景：同时维护投资人、客户、合作方关系，没有团队帮忙记录
- 痛点特征：关系密度高、全靠自己、对"下次怎么说"的需求极其具体
- 技术社区活跃度：高

**第三类：个人关系卡点**
- 典型场景：和父母、伴侣、重要朋友反复陷入同样的矛盾
- 痛点特征：情感共鸣强，传播性极高
- 付费意愿：低，但受众最广

### 4.2 为什么 MVP 阶段优先选第二类（独立顾问/创业者）

这个选择基于三个判断：

**第一，需求与技术社区高度重叠。** 独立顾问和创业者本身就活跃在 Hacker News 和 GitHub 上，他们用了觉得有用会自己发帖分享，这是最自然的技术社区增长路径。

**第二，输入质量最高。** 这类用户有习惯写复盘笔记，输入质量高，AI 输出也更稳定，能支撑一个有说服力的 demo。

**第三，痛点同时覆盖职场和个人。** 独立顾问/创业者的关键关系横跨客户、投资人、合作方，一个 hero demo 可以触达多类用户场景。

**关于第三类（家庭/个人关系）：** 情感共鸣更强，更容易 viral，但输入质量难以控制（目标模糊、情绪化输入），AI 容易产生"正确的废话"，会伤害产品口碑。这个方向作为后续迭代方向，不在 MVP 阶段引入。

## 5. 三个假想 Use Case

这三个案例覆盖了目标用户的核心场景，会出现在 README 的 demo 区域。

---

### Use Case A（Hero Demo）：独立顾问 vs 难搞客户

**背景：**
Sarah，独立产品顾问，服务一家传统制造业客户做数字化转型。对接人是运营总监王总，合作三个月，每次提方案都被否，但王总从不说具体原因。

**用户输入：**

> **这个人是谁：** 王总，某制造业公司运营总监，我的甲方对接人，合作三个月。
>
> **上次发生了什么：** 今天第四次提方案，王总又说"再想想"。但有个细节：上次我提到会影响现有团队的工作流程，他脸色明显变了。今天我刻意绕开这个点，他态度好一点了，但还是没拍板。他问了两次"别的公司怎么做的"，我感觉他需要外部参照，但我不确定。
>
> **下次我想达成什么目标：** 下次提案让他给出明确答复，不要再"再想想"。

**AI 输出：**（见第 6 节输出格式设计）

---

### Use Case B：创业者 vs 投资人

**背景：**
张明，独立创业者，在找 Pre-A 融资。上周和某 VC 合伙人李总开了第一次会，聊了 45 分钟，感觉没谈好，下周还有一次 follow up。

**用户输入：**

> **这个人是谁：** 李总，某 VC 合伙人，第一次见面，下周还有 follow up。
>
> **上次发生了什么：** 李总一开始很热情，但我一说到竞争对手他就明显冷淡下来，话变少了。我解释了半天我们的差异化，他好像没在听。后来我提到我们在东南亚有一个客户，他突然来了兴趣，问了很多细节。会议结束时他说"我们再看看"，感觉不太好。
>
> **下次我想达成什么目标：** 下次 follow up 能推进到 term sheet 讨论。

---

### Use Case C：职场中层 vs 强势老板

**背景：**
陈雷，某公司中层，需要推动一个新项目，但直属老板刘总一直不表态，既不反对也不支持，项目卡在原地三个月。

**用户输入：**

> **这个人是谁：** 刘总，我的直属老板，需要他拍板给资源才能推进项目。
>
> **上次发生了什么：** 今天又找刘总聊，他说"这个方向是对的"，但就是不给资源。我问他顾虑是什么，他说"你再完善一下方案"。这已经是第三版方案了。上次我带了数据他看得比较认真，但最后还是没有结论。我怀疑他有什么没说出来的顾虑，但我不知道是什么。
>
> **下次我想达成什么目标：** 让他真正表态，或者至少说清楚卡在哪里。

## 6. 输出格式设计

所有输入都会生成一张固定结构的 Playbook 卡片，六个区块。

以 Use Case A（Sarah vs 王总）为例：

---

### 区块① 这个人的沟通模式

> - 回避直接冲突，用"再想想"代替拒绝
> - 决策需要外部背书，"别人怎么做"是他的安全感来源
> - 对影响现有团队的变化有明显阻力，可能涉及内部政治

---

### 区块② Do（下次这样做）

> - 开场先给一个同行案例："XX 公司上个月做了类似的事，结果是..."
> - 把方案里涉及团队变动的部分最小化，或放到最后
> - 给他一个二选一，而不是开放式问题，降低他的决策成本

---

### 区块③ Don't（下次避开这些）

> - 不要再问"你的顾虑是什么"——他不会直接说
> - 不要一次提太多变化点，他需要感觉风险可控
> - 不要在他没表态时追问原因，会让他更防御

---

### 区块④ 下次可以直接用的话术

> 1. "我最近帮另一家类似规模的公司做了这个，他们的做法是..."
> 2. "我把方案分成两步，第一步完全不影响现有团队，你看这部分可以先推进吗？"
> 3. "这两个方向你倾向哪个？我按你的意思调整。"

---

### 区块⑤ 风险提醒

> 他的"再想想"可能不是在评估方案，而是在等内部某个人点头。建议下次直接问："这个决定除了你，还有谁需要参与？"

---

### 区块⑥ 置信度 + 待验证问题

> **置信度：** 中
>
> **待验证：** 下次用同行案例开场，观察他的反应是否明显比之前积极——如果是，说明外部背书是真实需求。如果没反应，说明阻力在别处，需要重新分析。

---

## 7. 产品设计原则

### 7.1 输入框设计原则

三个字段，不多不少：

- **字段一**：这个人是谁（一句话，定位关系）
- **字段二**：上次发生了什么（自由写，建议 200 字以内）
- **字段三**：下次我想达成什么目标（一句话，锁定目的）

提示语不是"描述你们的关系"，而是"上次发生了什么，你希望下次达成什么目标"——这把用户从"分析模式"切换到"备战模式"。

### 7.2 使用流程

```
谈完一次重要会面
    ↓
打开工具，填三个字段（会后复盘）
    ↓
得到一张 Playbook 卡片
    ↓
下次见面前打开复习（会前备战）
    ↓
见面后更新记录，持续迭代
```

**现有工具几乎只做了第一步（帮你记录），没人做第三步（帮你准备）。这是核心差异化。**

### 7.3 MVP 不做的事

- 不做联系人管理系统
- 不做用户登录和账号体系
- 不做数据持久化存储
- 不做聊天记录解析（第二阶段）
- 不做多 agent 架构（第二阶段）

---

## 8. 技术实现方案（MVP）

### 8.1 整体原则

一周内跑通，够用就好，不过度设计。

### 8.2 技术栈

| 层级 | 技术选型 | 理由 |
|---|---|---|
| 前端 | Next.js 单页应用 | 部署简单，开源贡献门槛低 |
| 后端 | Next.js API Route | 不需要单独起服务，一个 repo 搞定 |
| AI 调用 | Claude API（单模型） | MVP 阶段不做多 agent，一个精心设计的 prompt 足够 |
| 存储 | 无（浏览器内存） | 数据不离开用户本地，支撑隐私叙事 |
| 部署 | Vercel | 免费，自动部署 |

### 8.3 核心 Prompt 设计原则

- 强制输出六个区块的结构化结果（可用 JSON 或 Markdown 格式）
- 强制每条规则有行为证据支撑（引用输入里的具体细节）
- 强制给"置信度"和"待验证问题"——不确定的事情不瞎猜，让用户自己去验证

### 8.4 隐私叙事

> "所有数据只在你的浏览器里，不存储到任何服务器。开源代码可审计，也可以自己部署填入自己的 API key。"

这是对竞品的直接差异化，也是开源项目的天然优势。

### 8.5 一周开发计划

| 时间 | 任务 |
|---|---|
| Day 1–2 | Prompt 设计与调试，用三个 use case 反复跑，直到输出稳定 |
| Day 3–4 | 前端 UI，三字段输入 + 六区块输出卡片，卡片要好看 |
| Day 5 | 导出图片 / 一键复制话术功能 |
| Day 6 | 用真实案例跑通，写 README（含三个 demo） |
| Day 7 | 发布：GitHub + Hacker News + X/Twitter |

### 8.6 第二阶段可以加的技术亮点（MVP 之后）

这些是技术社区会讨论的内容，留给后续迭代：

- **多 agent 架构**：Pattern Agent / Risk Agent / Strategy Agent / Counterexample Agent + Editor Agent 综合输出
- **聊天记录解析**：支持微信/Slack 文字记录输入
- **规则持久化**：本地 JSON 文件存储，支持跨次迭代更新置信度
- **对方视角功能**：生成"对方眼中你的沟通风格"，viral 传播钩子

## 9. 影响力建立策略

### 9.1 技术社区（第一阶段）

**GitHub README 设计原则：**
- Hero demo 用案例 A（Sarah vs 王总），30 秒内让人明白这是什么
- 三个 use case 并排放，用户自己对号入座
- 技术架构有值得讨论的点（即使 MVP 是单模型，也要写清楚"为什么这样设计"和"下一步要做什么"）

**Hacker News 发帖叙事：**

> "I built an open-source tool that turns your messy relationship notes into an actionable communication playbook. Instead of 'this person is Type A', it tells you 'next time, open with a peer case study, then give them a binary choice'. Here's how it works."

**配套技术文章（发布后一周内写）：**

> 标题方向："为什么 AI 应该输出规则而不是总结——我做了一个实验"
> 内容：产品理念 + 技术选型理由 + 真实输出案例对比 + 下一步多 agent 的设计思路

### 9.2 大众传播（第二阶段迭代后）

引入家庭/个人关系场景，加入"对方视角"分享功能，走情感共鸣 + 截图传播路线。

## 10. 风险与需要提前规避的坑

**产品方向的坑：**
- 输出太长太泛，看完仍不知道下次怎么说 → 强制每个区块不超过 3 条
- 变成人格测试，用户觉得"玄学" → 每条规则必须引用输入里的具体行为证据
- 没有置信度，用户不信 → 强制输出置信度和待验证问题

**技术可信度的坑：**
- Prompt 约束不够，AI 过度推断稀薄证据 → MVP 阶段宁可输出"证据不足，待验证"也不要胡猜

**隐私的坑：**
- 用户担心职场/个人敏感信息上传到陌生服务器 → 明确说明"数据不离开浏览器"，开源可审计

## 11. 项目定位总结

| 维度 | 定位 |
|---|---|
| 本质 | 从"记录关系"升级为"指导关系" |
| 核心差异 | 输出可执行规则，不是泛泛洞察 |
| 目标用户（MVP）| 独立顾问 / 创业者，职场关键关系场景 |
| 输入（MVP）| 结构化复盘笔记（三字段） |
| 输出 | 六区块 Playbook 卡片 |
| 开发模式 | Indie，一周 MVP，持续迭代 |
| 开源策略 | 数据本地，可自部署，代码可审计 |
| 第一传播渠道 | GitHub + Hacker News |
| 成功指标（MVP）| 有人在 HN 或 X 上转发真实使用体验 |
